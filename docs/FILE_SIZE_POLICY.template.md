# File Size Policy (파일 크기 및 분할 규칙)

* **사용 목적**: 소스 파일이 거대해져 AI 에이전트의 컨텍스트 해상도가 떨어지고 오작동 루프를 도는 현상을 방지합니다.
* **작성자**: Documenter / Technical Evaluator
* **포함 사항**: 최대 파일 라인 수 기준, 강제 분할 절차, 리팩토링 동의 요건.

---

## 1. 파일 크기 경고 및 강제 분할 임계치 (Thresholds)
하네스는 에이전트가 편집하는 파일들의 크기를 엄격히 제어하며, `harness.config.json` 설정을 준수합니다.
* **경고 및 감사 대상 (Warning)**: 단일 코드 파일이 **300줄** 또는 **12KB**를 초과할 때.
* **강제 분할 분기점 (Hard Limit)**: 단일 코드 파일이 **500줄** 또는 **20KB**에 도달한 경우.
* **검출 도구**: `scripts/file_size_audit.ps1` 및 `scripts/file_size_audit.sh`를 활용해 상시 자동 검사.

## 2. 파일 분할 절차 (Safe Split Steps)
파일 크기가 임계치를 초과한 경우 구현자는 즉시 코딩을 멈추고 다음 단계의 안전 분할 계획을 수립해야 합니다:
1. **관심사 분리 (Separation of Concerns)**: 거대 파일 내부에서 하위 컴포넌트, 유틸리티 헬퍼 함수, 데이터 타입 명세를 분리할 후보군을 식별합니다.
2. **계획 문서 작성**: `system/decision_log.md`에 분할할 파일 목록과 의존 관계 변화를 먼저 기록합니다.
3. **수동 승인 요청**: 아키텍처에 파급력이 크므로 반드시 사용자(Human)의 사전 승인을 구합니다. (동의 없이 다수 파일을 임의 분할 금지)
4. **리팩토링 이행**: 모듈을 점진적으로 추출하고 기존 파일의 참조 코드를 수정합니다.
5. **검증**: 컴포넌트가 정상 빌드 및 렌더링되는지 확인합니다.

## 3. 예외 조항 (Exceptions)
* 구조상 분할이 불가능한 대형 외부 라이브러리 코드나 정적 데이터 정의 파일의 경우, `harness.config.json` 예외 목록에 등록하여 진단 대상에서 배제할 수 있습니다.
