# Bug Reproduction Policy (버그 재현 및 검격 규칙)

* **사용 목적**: 버그 재현 없이 짐작으로 패치 코드를 적용했다가 또 다른 버그를 만들고 무한 루프를 도는 실패를 방지합니다.
* **작성자**: Bug Reproducer / Technical Implementer
* **포함 사항**: 재현 실패 시 정지 룰, 예상/실제 결과 서식, 독립 테스트 구동 지침.

---

## 1. 재현 의무 선언 (Reproduction Rule)
* **원칙**: 모든 결함 수정 요청에 대해 구현자는 코드를 수정하기 전에 반드시 버그 발생 상황을 먼저 "로컬 환경에서 직접 관찰 및 재현"해야만 합니다.
* **예외**: 단순 오탈자 수정, 변수명 정합성 변경 같은 사소한 문법적 예외 작업.

## 2. 재현 실패 시 행동 강령 (If Not Reproducible)
* 3회 이상의 환경 구성 및 주입 시도 후에도 버그가 로컬에서 재현되지 않는 경우:
  - **행동**: 코드를 절대 패치하지 말고 수정을 중단하십시오.
  - **대처**: `system/issue_log.md`에 재현 미비 사유를 기입하고 즉시 사용자(Human)에게 버그 발생 재현 조건에 대한 추가 정보 또는 영상/로그 증적을 요청하십시오.

## 3. 재현 명세 서식 (Reproduction Template)
재현 성공 시 `system/issue_log.md`에 기재해야 하는 필수 사항입니다:
* **재현 경로**: (예: 페이지 접속 -> 모달 오픈 -> 입력창 빈칸 상태로 제출 클릭)
* **예상 결과 (Expected)**: (예: 폼 에러 메시지가 인라인으로 나타나야 함)
* **실제 결과 (Actual)**: (예: 에러가 나지 않고 빈 객체 상태로 폼이 제출되어 화면 크래시 발생)
* **재현 코드/로그**: (재현에 사용된 스크립트 또는 관련 콘솔 에러 로그 캡처)
