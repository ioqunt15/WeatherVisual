# Frontend Architecture (프론트엔드 아키텍처 규칙)

* **사용 목적**: 컴포넌트 구조화 규칙, 스타일링 가이드라인, 전역 상태 범위의 표준을 제시하여 확장과 유지보수를 쉽게 합니다.
* **작성자**: Frontend Implementer / Technical Evaluator
* **포함 사항**: 디렉토리 구조 예시, 스타일 관리 전략, 컴포넌트 분리 정책.
* **제외 사항**: 특정 프레임워크 전용 문법 소스 코드.

---

## 1. 컴포넌트 계층 정의 (Component Layering)
* **공통 UI 컴포넌트 (Shared/Common UI)**:
  - 오직 프레젠테이셔널 역할만 수행하며, 내부에서 비즈니스 로직 및 API 연동을 철저히 배제합니다. (예: 버튼, 입력창, 모달 뼈대)
* **도메인 컴포넌트 (Domain Components)**:
  - 특정 비즈니스 상태와 연결되어 실질적인 연산을 처리하는 화면 조각입니다.
* **페이지 컴포넌트 (Pages/Screens)**:
  - 최상위 라우팅 단위 화면으로, 하위 컴포넌트들을 조합하고 데이터를 주입하는 컨테이너 역할을 합니다.

## 2. 스타일링 원칙 (Styling Guidelines)
* 임의의 인라인 스타일링(`style={{ marginTop: '15px' }}`) 사용을 전면 배제합니다.
* CSS 클래스 변수를 재사용하거나 사전에 선언된 디자인 시스템 스페이싱 값을 사용하여 간격을 제어합니다.
* 스타일을 위한 컴포넌트 중첩(Wrapper div 과다 중첩)을 지양하고, 시맨틱 태그(article, section, main 등)를 알맞게 구조화하여 CSS 그리드나 플렉스를 적용합니다.

## 3. 상태 관리 지침 (State Management Rules)
* **KISS / YAGNI 원칙**: 
  - 단순 컴포넌트 내부 상태(`useState` 등)로 처리 가능한 일시적인 UI 상태를 Redux, Zustand 등의 전역 스토어에 억지로 이관하여 오버엔지니어링을 유발하지 마십시오.
  - 전역 스토어는 오직 다수의 페이지가 공유하는 사용자 세션, 전역 테마 상태 등에 한해서만 엄선하여 설계합니다.
