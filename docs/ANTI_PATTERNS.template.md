# AI Anti-Patterns (AI 에이전트 실패 방지 가이드)

* **사용 목적**: AI 에이전트들이 반복적으로 저지르는 코딩/디자인 실수 패턴을 명시하고, 이를 자가 진단 및 예방하는 지침서입니다.
* **작성자**: All Roles (특히 Technical/Design Evaluator)
* **포함 사항**: 6가지 흔한 실패 패턴 분석 및 예방법.

---

## 1. 흔한 AI 실패 패턴 및 대책 (Anti-Patterns)

### A. 시각적 무변화 수정 (Ghost Commits)
* **패턴**: 코드는 왕창 수정했는데 정작 브라우저 상의 비주얼이나 레이아웃은 단 1픽셀도 개선되지 않은 상태.
* **대책**: 시각적 수정 전에 목표 결과를 정확히 글이나 스크린샷 가이드로 명시하고, 수정 완료 후 시각 변화가 발생했는지 디자인 게이트를 통해 즉시 증명합니다.

### B. 짐작으로 패치하기 (Blind Patching)
* **패턴**: 버그 보고를 접한 후, 로컬에서 문제 상황을 재현하거나 검증하지 않고 짐작되는 부분을 땜질해놓고 "수정했다"고 주장하는 것.
* **대책**: 수정 전 재현 절차를 통과하지 않은 버그는 완료 처리될 수 없습니다. `bug_reproduction_before_patch.md` 워크플로우를 절대 준수해야 합니다.

### C. 개성 없는 템플릿 UI 남발 (Generic Library UI)
* **패턴**: 목적에 맞춘 UI 디자인 대신, 자신이 아는 기본 부트스트랩이나 Tailwind 카드 예제를 그대로 배치하고 무분별한 그라데이션, 배지, 그림자를 남발해 촌스러운 UI를 만드는 것.
* **대책**: 비주얼 디자인 디렉터가 정의한 `VISUAL_SYSTEM.md` 규칙을 반드시 따라야 합니다. 필요 없는 장식 요소는 일체 제거합니다.

### D. 관련 없는 코드 과도 수정 (Over-Editing)
* **패턴**: 하나의 조그만 버그를 고치기 위해 수동 승인 없이 유틸리티 모듈 전체나 다른 소스 파일 수십 개를 난도질해 놓아 버그를 전파시키는 것.
* **대책**: 수정 대상 영역의 파일에 한해서만 락(Lock)을 설정하고 편집 범위를 좁힙니다.

### E. 무한 수정 루프 (Endless Self-Repair Loop)
* **패턴**: 에러를 해결한답시고 수정했다가 다른 에러가 터지고, 이를 스스로 고치느라 의미 없는 수정 루프를 10회 이상 돌며 토큰을 낭비하는 현상.
* **대책**: 수정 ↔ 평가 루프는 **최대 3회**로 강제 제한되며, 3회 실패 시 반드시 멈추고 휴먼 게이트를 트리거해야 합니다.

### F. 성급한 DRY 원칙 적용 (Premature DRY Abstraction)
* **패턴**: 사소한 중복 코드를 없애겠다고 추상화 계층을 만들었다가 코드 가독성이 극도로 악화되고 수정이 어려워지는 오버엔지니어링.
* **대책**: KISS 및 YAGNI 원칙이 DRY 원칙보다 우선합니다. 명확성을 위해 약간의 중복은 인정할 수 있어야 합니다.
